tmux + iTerm2 +
Claude Code Workflow

Persistent terminal sessions across the macOS fleet. Sessions survive disconnects. Logs sync via iCloud. Native iTerm2 look and feel.

macbook-pro macbook-air mac-pro

How it works

Every iTerm2 window automatically connects to a tmux session in control mode (tmux -CC). iTerm2 renders tmux windows as native tabs and tmux panes as native splits. You get persistent sessions with the full iTerm2 experience: scrollback, search, native copy/paste, font rendering.

~/.zshrc
# Auto-launch tmux in iTerm2 control mode (native tabs/splits)
if [[ -z "$TMUX" && "$TERM_PROGRAM" == "iTerm.app" && -z "$INSIDE_EMACS" && -z "$VSCODE_PID" ]]; then
  tmux -CC new-session -A -s main
fi
  • -CC iTerm2 control mode — native tabs and splits instead of terminal-in-terminal
  • -A attach to existing session if it exists, create if it doesn't
  • -s main session name
  • Guards prevent nesting, non-iTerm terminals, and editor shells

Session logging

All pane output is automatically logged to ~/Documents/logs/ (iCloud-synced) via tmux hooks in ~/.tmux.conf.

Log filename format
hostname-YYYY-MM-DD.HH:MM:SS.log
Example logs
macbook-pro-2026-03-21.14:03:22.log
mac-pro-2026-03-21.09:15:00.log
macbook-air-2026-03-21.11:30:45.log

New windows automatically start logging. Logs from all three machines sync to iCloud via ~/Documents/logs/.

Day-to-day usage

Local (sitting at the machine)

Open iTerm2. It auto-connects to the main tmux session. If you had windows open before, they're all still there.

ActionShortcut
New windowCmd+T
Split horizontalCmd+D
Split verticalCmd+Shift+D
Close paneCmd+W or exit
Switch tabsCmd+1-9 or Cmd+Left/Right

All standard iTerm2 shortcuts work because iTerm renders the tmux session natively.

Remote (SSH between machines)

SSH to another machine and attach to its tmux session. iTerm2 opens the remote session in control mode — remote tabs appear as native iTerm tabs. If SSH drops, the session keeps running. Reconnect and reattach. 

# From macbook-pro, attach to mac-pro's session
ssh mac-pro -t "tmux -CC attach -s main"

# If no session exists yet
ssh mac-pro -t "tmux -CC new-session -A -s main"

Claude Code in tmux

Claude Code works normally inside tmux sessions. If a long task is running and your connection drops, the tmux session keeps it alive. Reattach and it's still going. 

# Tab 1: Claude Code session
cd ~/Documents/projects/cosmicllama
claude

# Tab 2: dev server or tests (Cmd+T for new tab)
npm run dev

# Tab 3: SSH to another machine
ssh mac-pro
  • Terminal detection — Claude Code detects the terminal correctly through tmux
  • Shell integration — iTerm2 marks and tab titles work as expected
  • Passthrough — The allow-passthrough setting in .tmux.conf ensures iTerm2 escape sequences pass through

Named sessions

The default session is main. Create purpose-specific sessions when needed. 

# Start a dedicated session for a project
tmux -CC new-session -s cosmicllama

# List sessions
tmux list-sessions

# Attach to a specific session
tmux -CC attach -s cosmicllama

Key bindings

These work in regular tmux mode (without -CC). In control mode, iTerm2 shortcuts take over, but prefix keys (Ctrl+B or Ctrl+A) still work.

BindingAction
prefix + |Split horizontally
prefix + -Split vertically
prefix + cNew window
prefix + h/j/k/lNavigate panes (vim-style)
prefix + H/J/K/LResize panes
prefix + rReload config

Troubleshooting

Common issues and how to resolve them.

⚠️

"sessions should be nested with care"

You tried to start tmux inside tmux. The .zshrc guard prevents this automatically, but if you manually run tmux inside a session, use tmux -CC attach -t other-session instead.

🔌

Detached session won't reattach

Kill stale sessions with tmux kill-session -t name and start fresh. If tmux list-sessions shows the session but attach fails, the control mode client may have hung. Kill and reconnect.

📝

Logs not appearing

Check that ~/Documents/logs/ exists. New windows trigger the logging hook; existing windows from before the config change won't log until you open a new one.

🤖

SSH dropped mid-Claude

SSH back in and run tmux -CC attach -s main. Your Claude Code session is still running. The tmux session kept it alive on the remote machine.
